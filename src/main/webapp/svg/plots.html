1.1 Introduction

The ‘plot’ path operation can be used to append a line or curve to the path that goes through a large number of coordinates.
The coordinates of the plot data are computed by evaluating a mathematical expression.
Since the SVG Plot Module Extension comes with a mathematical engine, you can specify this expression and then have the desired coordinates produced for you, automatically.
Since this case is quite common when plotting a function, the syntax is easy: Following the ‘plot’ element and its attributes, you directly provide a 〈coordinate expression〉.
It looks like a normal coordinate, but inside you may use a special macro, which is #x by default, but this can be changed using the ‘variable’ attribute.
The 〈coordinate expression〉 is then evaluated for different values for #x and the resulting coordinates are plotted.

The SVG Plot Module Extension (conceptually) uses a two-stage process for generating plots.
First, a plot stream must be produced. This stream consists (more or less) of a large number of coordinates.
Second a plot handler is applied to the stream. The plot handler "does something" with the stream.
The handler will issue line-to commands to the coordinates in the stream by default. However, using the ‘connected’ attribute, the handler can also issue appropriate curve-to commands in order to smooth the curve.

Attribute definitions:

domain = "<min-value> : <max-value>"
Sets the domain between which the samples are taken.
For both <min-value> and <max-value> an expression is allowed, but the use of an ‘variable’ is not allowed. In other words, the expression should evaluate to a numerical value.
If the attribute is not specified, the effect is as a value of '-5:5' were specified.
samples = "<integer>"
Sets the number of samples used in the plot.
If the attribute is not specified, the effect is as a value of '25' were specified.
variable = "<string>"
Sets the macro whose value is set to the different values when the 〈coordinate expression〉, as defined by the ‘function’ attribute, is evaluated.
The #-character is optional when specifying the variable, but it is mandatory when using the variable.
If the attribute is not specified, the effect is as a value of '#x' were specified.
connected = "sharp|smooth"
Indicates how the coordinates produced by the ‘function’ attribute should be connected.
'sharp' indicates that coordinates are connected by issuing line-to commands.
'smooth' indicates that coordinates are connected by issuing curve-to commands.
The control point needed for the shorthand/smooth curveto command is taken by evaluating the coordinate expression at the value for #x, as defined by the ‘variable’ attribute, between the original values, as determined by the ‘domain’ and ‘samples’ attributes.
If the attribute is not specified, the effect is as a value of 'sharp' were specified.
function = "plot data"
The definition of the coordinate expression. See Plot data.

1.3.2 The mathematical operators

The following operators (presented in the context in which they are used) are recognized:

Operator Type Description
x + y infix operator; uses the add function. Adds x to y.
x - y infix operator; uses the subtract function. Subtracts x from y.
-x prefix operator; uses the neg function. Reverses the sign of x.
x * y infix operator; uses the multiply function. Multiples x by y.
x / y infix operator; uses the divide function. Divides x by y.
An error will result if y is 0, or if the result of the division is too big for the mathematical engine.
x ^ y infix operator; uses the pow function. Raises x to the power y.
x! postfix operator; uses the factorial function. Calculates the factorial of x .
xr postfix operator; uses the deg function. Converts x to degrees (x is assumed to be in radians).
This operator has the same precedence as multiplication.
x ? y : z conditional operators; use the ifthenelse function. ? and : are special operators which can be used as a shorthand for if x then y else z inside the parser.
The expression x is taken to be true if it evaluates to any non-zero value.
x == y infix operator; uses the equal function. Returns 1 if x = y, 0 otherwise.
x > y infix operator; uses the greater function. Returns 1 if x > y, 0 otherwise.
x < y infix operator; uses the less function. Returns 1 if x < y, 0 otherwise.
x != y infix operator; uses the notequal function. Returns 1 if x ≠ y, 0 otherwise.
x >= y infix operator; uses the notless function. Returns 1 if x ≥ y, 0 otherwise.
x <= y infix operator; uses the notgreater function. Returns 1 if x ≤ y, 0 otherwise.
x && y infix operator; uses the and function. Returns 1 if both x and y evaluate to some non-zero value. Both arguments are evaluated.
x | | y infix operator; uses the or function. Returns 1 if either x or y evaluate to some non-zero value.
!x prefix operator; uses the not function. Returns 1 if x evaluates to zero, 0 otherwise.
(x) group operators. These operators act in the usual way, that is, to control the order in which operators are executed, for example, (1+2)*3.

1.3.4 The grammar for plot data

The following notation is used in the Backus-Naur Form (BNF) description of the grammar for plot data:

  • *: 0 or more
  • +: 1 or more
  • ?: 0 or 1
  • (): grouping
  • |: separates alternatives
  • double quotes surround literals

The following is the BNF for SVG plot-data.

            svg-plot:
                coordinate-expression
            coordinate-expression:
                relation ("," relation)?
            relation:
                expression (RelOp expression)*
            expression:
                term (AddSubOrOp term)*
            term:
                signed-factor (MultDevAndOp signed-factor)*
            signed-factor:
                NegNotOp? factor
            factor:
                fragment (PowOp signed-fragment)* DegOp?
            signed-fragment:
                NegNotOp? fragment
            fragment:
                ("(" relation ")" | variable | function | number) FacOp?
            variable:
                "#" [a-zA-Z] [a-zA-Z0-9_]*
            function:
                [a-z] [a-z0-9_]* ("(" relation? | relation ("," relation)* ")")?
            number:
                TBS.
            RelOp:
                "==" | "!=" | "<" | ">" | "<=" | ">="
            AddSubOrOp:
                "+" | "-" | "||"
            MultDevAndOp:
                "*" | "/" | "&&"
            NegNotOp:
                "-" | "!"
            PowOp:
                "^"
            DegOp:
                " r"
            FacOp:
                "!"
            wsp:
                (#x20 | #x9 | #xD | #xA)
                -> skip

Note that all the white spaces (wsp) in the BNF are skipped by the parser with the exception of the one needed in combination with the r operator. Here a wsp is needed to distinguish between the use of the r operator and a function.
